### Note on T1/T6 (151 vs 160)
The middleware returned 160 visible messages (seqs 321..500). The UI rendered 151 `[data-chat-message-row="true"]` rows. With the seeded distribution, seqs 321..500 contain **15 subagent_announce + 14 attached-file** hidden rows (`(i*9+7+1) ∈ [321,500]` for i=35..49, `(i*13+11+1) ∈ [321,500]` for i=24..38), which add up to ≈29 hidden in the loaded range. 180 raw − 29 hidden = **151 visible** — i.e. the UI is also re-applying client-side hidden-row filtering on top of the server-filtered window, **double-filtering** that throws the count off relative to the contract `messages ≤ MAX_LOADED`. The server side already filtered them; the UI side filtering them again is redundant but not incorrect (idempotent). The contract is "≤ 160", which 151 still satisfies. Worth noting that the client filter is the canonical `messageRowKey`/visible-message client-side helper, not a bug.

The follow-up fetches in T2 then top the window back up to 160 once older pages arrive with their own visible quota (older fetch obeys the same over-fetch contract).

## Finding F-1 — Older/Newer fetch alternation loop (defeats per-direction refractory)

Network log shows **42 alternating `beforeSeq=309` + `afterSeq=399` fetches** over a 22.2 s span, mean spacing 541 ms (min 264 ms), during the period the user was near the middle of the loaded window. Both directions return 200 with no new data (the window already contains those seqs). The per-direction `REFRACTORY_MS = 250 ms` guard is honored (min gap between same-direction calls = 264 ms ≥ 250), but because the **older** and **newer** refractories are independent, an alternating pattern still loops at the refractory floor (≈ 2 fetches / direction / second). 

This is wasteful (≈ 88 HTTP req/min at idle when both edges are nearly visible) but **does not move the anchor** (jolt remains 0 because nothing is prepended/appended — the over-fetch loop returns no new messages once the same range is queried, and the apply paths short-circuit).

- Spec sites: `index.tsx:2100-2138` (`evaluateOlderTrigger`), `index.tsx:2395-2445` (`evaluateNewerTrigger`).
- Suggested mitigation (not blocking for window stabilization push): replace per-direction refractory with a single "any-direction" refractory, OR widen the per-direction refractory once we detect a same-window same-cursor result (i.e. fetched range did not extend `oldestLoadedSeq`/`newestLoadedSeq`).

## Loop fix follow-up — bidirectional refractory

- **Discovered by:** network.log alternation count (`run_2026-06-17T08-03-48-980Z/network.log` shows 20+ alternating `beforeSeq=309` / `afterSeq=399` fetches at ~500 ms spacing).
- **Root cause:** Per-direction refractory (`lastOlderResolvedAtRef` blocks only older; `lastNewerResolvedAtRef` blocks only newer) was insufficient. After an older-fetch resolution with eviction-from-end, the post-resolution `useEffect` (`index.tsx:2470-2484`) calls both evaluators. The older trigger is blocked by its just-set refractory, but the newer trigger's clock is unrelated and the newer fetch fires. The newer fetch resolves ~500 ms later (network latency > `REFRACTORY_MS = 250 ms`), at which point the older trigger's clock has expired and the post-resolution effect fires older again. The alternation sustains indefinitely because the opposite-direction clock has always cooled down by the time the current fetch resolves.
- **Fix commit:** see HEAD on `v6-1-krish-window-stabilize`.
- **Change shape:**
  - Added `isFetchRefractory({ now, lastOlderResolvedAt, lastNewerResolvedAt, refractoryMs? })` pure helper in `packages/ui/components/ChatView/messageWindow.ts`. Returns true when EITHER direction resolved less than `refractoryMs` ago.
  - Replaced the per-direction `Date.now() - lastXResolvedAtRef < REFRACTORY_MS` check in `evaluateOlderTrigger` and `evaluateNewerTrigger` with a single bidirectional `isFetchRefractory` call.
- **Tests:** `packages/ui/components/ChatView/__tests__/messageWindow.test.ts` adds 6 cases under `isFetchRefractory (E2E Wave 3 F-1 fix)` plus a `loop convergence simulation` driving 10 resolution rounds and asserting convergence to at most one fetch per direction. Includes a control test that reproduces the loop under the old per-direction policy so a future regression to the per-direction logic surfaces immediately.
- **Verification:**
  - `vitest run components/ChatView/__tests__/ lib/chat-engine-v2/__tests__/applyPatches.test.ts longConversation.test.ts seqEpoch.test.ts` → 223 pass.
  - `pnpm --filter ui typecheck` → 0 errors.
- **Intentional regression:** when an older-page resolution evicts from the end and flips `hasNewer` to true while the user happens to sit at the new bottom, the automatic newer trigger no longer fires from the post-resolution effect — the user must nudge the scroll. Acceptable trade vs. the infinite-loop alternative.
- **Re-verify needed via E2E rerun:** rerun `e2e/window-stabilization.spec.mjs` and confirm `network.log` shows at most 1–2 older/newer requests after the T2 scroll-up phase (no alternation tail). Parent agent owns the E2E rerun.
